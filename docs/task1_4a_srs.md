## 1. Введение

### 1.1 Цель

Документ описывает требования к учебной desktop-системе учета волонтерской помощи. SRS используется как основание для проектирования, реализации, тестирования и приемки прототипа.

### 1.2 Область применения

Система поддерживает локальную работу координатора: прием заявок, ведение справочника волонтеров, назначение волонтера, контроль статусов, просрочки, погодные предупреждения и уведомления.

### 1.3 Определения

| Термин | Значение |
| --- | --- |
| Заявка | Запись о потребности заявителя в волонтерской помощи |
| Волонтер | Исполнитель, которого можно назначить на заявку |
| Координатор | Пользователь, распределяющий заявки между волонтерами |
| Просрочка | Заявка в статусе «Новая» без волонтера более 3 дней |
| Погодное предупреждение | Текстовое предупреждение для уличной заявки |

## 2. Общее описание

### 2.1 Вид продукта

Приложение является учебной desktop-системой на Java 17 + JavaFX 17. Данные хранятся локально в JSON-файле.

### 2.2 Пользователи

Система ориентирована на волонтера, координатора и администратора/оператора как непосредственных пользователей desktop-интерфейса. Заявитель является внешним источником данных: его сведения вводит оператор или координатор. В учебной реализации роли не разделены авторизацией, но их действия отражены в интерфейсе и сценариях.

### 2.3 Ограничения

- Реальная SMS-отправка заменена формированием текста сообщения.
- При отсутствии API-ключа используется учебная заглушка погоды.
- Хранение локальное, без серверной базы данных.
- Авторизация и разграничение прав не реализованы в первой версии.

### 2.4 Допущения

- Координатор работает на локальном компьютере.
- Данные заявителя вводятся оператором или координатором.
- Плановая дата хранится в формате `yyyy-MM-dd HH:mm`.

## 3. Функциональные требования

| Код | Описание | Приоритет | Критерий проверки |
| --- | --- | --- | --- |
| FR-01 | Система должна позволять создать заявку с ФИО, телефоном, типом помощи, типом места, адресом, датой, временем и описанием. | Must | После сохранения заявка отображается в таблице. |
| FR-02 | Система должна отображать список всех заявок в табличной форме. | Must | Вкладка «Заявки» содержит таблицу с заявками. |
| FR-03 | Пользователь может открыть карточку выбранной заявки для просмотра подробной информации. | Must | Кнопка «Подробно» открывает карточку выбранной заявки. |
| FR-04 | Система должна позволять редактировать заявку, если она не завершена и не отменена. | Must | Измененные поля сохраняются для открытой заявки. |
| FR-05 | Система должна позволять отменить заявку с указанием причины отмены. | Must | Заявка получает статус «Отменена» и причину отмены. |
| FR-06 | Система должна автоматически присваивать новой заявке статус «Новая». | Must | У созданной заявки установлен статус «Новая». |
| FR-07 | Система должна позволять координатору назначить доступного волонтера на заявку. | Must | В заявке сохраняется идентификатор выбранного волонтера. |
| FR-08 | Система должна автоматически переводить заявку в статус «В работе» после назначения волонтера. | Must | После назначения в таблице отображается статус «В работе». |
| FR-09 | Пользователь может отметить заявку в работе как выполненную. | Must | Кнопка «Выполнить» доступна для заявки в работе. |
| FR-10 | Система должна записывать статус «Выполнена» и дату завершения после выполнения заявки. | Must | В карточке заявки есть статус «Выполнена» и дата завершения. |
| FR-11 | Система должна проверять заявки в статусе «Новая» без волонтера, созданные более 3 дней назад. | Must | Проверка просрочек находит только новые заявки без волонтера старше 3 дней. |
| FR-12 | Система должна автоматически переводить найденные заявки в статус «Просрочена». | Must | Найденная заявка получает статус «Просрочена». |
| FR-13 | Система должна формировать адресное сообщение координатору при просрочке заявки. | Must | В списке уведомлений появляется сообщение по просроченной заявке. |
| FR-14 | Система должна проверять погоду для заявки с типом места «Улица». | Must | Для уличной заявки вызывается погодный сервис или заглушка. |
| FR-15 | Система должна добавлять погодное предупреждение к заявке при неблагоприятных погодных условиях. | Must | В карточке заявки отображается текст погодного предупреждения. |
| FR-16 | Система должна позволять добавить волонтера в справочник. | Should | Новый волонтер появляется во вкладке «Волонтеры». |
| FR-17 | Система должна позволять редактировать данные волонтера. | Should | Измененные ФИО, телефон, навыки или доступность сохраняются. |
| FR-18 | Система должна позволять удалить волонтера, если он не назначен на активную заявку. | Should | Неназначенный волонтер удаляется, назначенный на активную заявку не удаляется. |
| FR-19 | Система должна вести историю изменения статуса заявки. | Should | В карточке заявки отображается история операций со статусами. |
| FR-20 | Пользователь может фильтровать заявки по статусу. | Should | Выбор статуса изменяет список заявок. |
| FR-21 | Пользователь может фильтровать заявки по типу помощи. | Should | Выбор типа помощи изменяет список заявок. |
| FR-22 | Пользователь может фильтровать заявки по плановой дате. | Should | Выбор даты изменяет список заявок. |
| FR-23 | Пользователь может фильтровать заявки по назначенному волонтеру. | Should | Выбор волонтера изменяет список заявок. |
| FR-24 | Система должна отображать счетчики заявок по статусам. | Should | Вкладка «Статистика» показывает счетчики по статусам. |
| FR-25 | Система должна загружать демонстрационные данные при первом запуске, если файл хранения еще не создан. | Could | При пустом хранилище появляются демонстрационные заявки и волонтеры. |
| FR-26 | Система должна сохранять заявки, волонтеров и уведомления в локальное JSON-хранилище. | Must | После изменения данных обновляется файл `data/app-data.json`. |

## 4. Нефункциональные требования

| Код | Требование | Критерий проверки |
| --- | --- | --- |

| NFR-02 | Интерфейс должен быть русскоязычным. | Все основные подписи интерфейса на русском языке. |
| NFR-03 | Приложение должно быть реализовано на Java 17 с использованием JavaFX. | `pom.xml` содержит Java 17 и JavaFX 17. |
| NFR-04 | Сборка проекта должна выполняться через Maven. | `mvn test` компилирует проект и запускает тесты. |
| NFR-05 | Код должен быть размещен в GitHub-репозитории при сдаче работы. | Репозиторий содержит `pom.xml`, `src/`, `docs/` и README. |

| NFR-10 | Основные бизнес-правила должны быть покрыты unit-тестами. | JUnit-тесты проверяют создание, назначение, просрочку и уведомления. |

## 5. Внешние интерфейсы

### 5.1 Пользовательский интерфейс

Главное окно содержит вкладки:

- «Заявки»;
- «Волонтеры»;
- «Назначение»;
- «Уведомления»;
- «Статистика».

Используются JavaFX-компоненты `TableView`, `TextField`, `ComboBox`, `DatePicker`, `Dialog`, `TabPane`, `Button`, `TextArea`.

### 5.2 Программные интерфейсы

- `WeatherService` обращается к OpenWeatherMap API при включенной настройке `weather.api.enabled=true`;
- `NotificationService` формирует текст сообщения координатору;
- `DataStore` читает и записывает JSON-файл.

### 5.3 Интерфейс хранения

Хранилище: `data/app-data.json`.

Сохраняемые группы данных:

- `requests`;
- `volunteers`;
- `notifications`.

Внутри элемента `requests` хранятся встроенные данные заявки: `weatherWarning`, `statusHistory` и `notificationText`. Отдельное физическое хранилище для истории статусов в первой версии не используется.

## 6. Критерии приемки

| Код | Критерий |
| --- | --- |
| AC-01 | Пользователь создает заявку, и она появляется в таблице со статусом «Новая». |
| AC-02 | Координатор назначает доступного волонтера, и статус заявки становится «В работе». |
| AC-03 | Пользователь завершает заявку в работе, и она получает статус «Выполнена». |
| AC-04 | Пользователь отменяет заявку и сохраняет причину отмены. |
| AC-05 | Проверка просрочек переводит заявку старше 3 дней без волонтера в статус «Просрочена». |
| AC-06 | При просрочке создается сообщение координатору и сохраняется в списке уведомлений. |
| AC-07 | Для уличной заявки система выполняет проверку погоды и при риске добавляет предупреждение. |
| AC-08 | Фильтры по статусу, типу помощи, дате и волонтеру изменяют список заявок. |
| AC-09 | После перезапуска данные загружаются из JSON-файла. |
| AC-10 | JUnit-тесты бизнес-логики проходят успешно в окружении с Maven. |
