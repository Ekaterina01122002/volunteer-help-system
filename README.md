# Пояснительная записка к курсовому проекту

Проект: ПС «Организация волонтерской помощи»

Источник шаблона: [CourseProject.report.md](https://github.com/olgmina/SWEngineering-technics.github.io/blob/gh-pages/Labs/CourseProject.report.md)

## Введение

Актуальность проекта связана с тем, что небольшие волонтерские центры часто обрабатывают заявки через таблицы, мессенджеры и устные договоренности. Такой подход усложняет контроль статусов, поиск свободных волонтеров и отслеживание заявок, которые долго остаются без исполнителя.

Цель курсового проекта: разработать учебное desktop-приложение для учета заявок на волонтерскую помощь, назначения волонтеров, контроля просрочек, проверки погодных условий и формирования уведомлений координатору.

Для достижения цели решены задачи:

1. Определены роли пользователей и сценарии использования.
2. Сформулированы функциональные и нефункциональные требования.
3. Подготовлена SRS-спецификация.
4. Построены DFD-диаграммы и CRUD-матрица.
5. Реализован прототип на Java 17 + JavaFX + Maven.
6. Подготовлены unit-тесты бизнес-логики.

## 1. Функциональное назначение программного средства

### 1.1 Классы решаемых задач

Программное средство решает следующие классы задач:

- задачи учета: хранение заявок, волонтеров и уведомлений;
- задачи поиска и фильтрации: поиск заявок по статусу, типу помощи, дате, волонтеру и тексту;
- задачи контроля: проверка статусов, просрочек и обязательных полей;
- задачи автоматизации: смена статуса после назначения и завершения;
- задачи информирования: погодные предупреждения и сообщения координатору.

Концепция программы представлена в `docs/task1_1_concept.md`.

### 1.2 Функциональное назначение

Программа предназначена для локальной работы координатора волонтерского центра. Пользователь может создать заявку, отредактировать ее, отменить, назначить волонтера, завершить работу, проверить просрочки, сформировать уведомление координатору, вести справочник волонтеров и просматривать статистику.

Сценарии использования представлены в `docs/task1_2_use_cases.md`.

### 1.3 Нефункциональные требования, ограничения и интерфейсы

Основные ограничения:

- учебная версия не выполняет реальную SMS-отправку;
- авторизация пользователей не реализована;
- данные хранятся локально в JSON-файле;
- при отсутствии API-ключа OpenWeatherMap используется заглушка.

SRS-спецификация представлена в `docs/task1_4a_srs.md`.

## 2. Проектная часть

### 2.1 Архитектура приложения

Выбрана простая модульная desktop-архитектура:

- `model` — сущности предметной области;
- `service` — бизнес-логика;
- `storage` — локальное JSON-хранилище;
- `ui` — JavaFX-интерфейс;
- `util` — вспомогательные функции дат и идентификаторов.

Такой вариант подходит для учебного проекта, потому что сохраняет разделение ответственности без избыточной серверной инфраструктуры.

### 2.2 Проектирование пользовательского интерфейса

Интерфейс реализован как JavaFX-приложение с вкладками:

- «Заявки»;
- «Волонтеры»;
- «Назначение»;
- «Уведомления»;
- «Статистика».

На вкладке «Заявки» используются таблица, поиск, фильтры, кнопки добавления, редактирования, отмены, завершения, просмотра карточки и проверки просрочек. Формы ввода реализованы через JavaFX-диалоги.

### 2.3 Модель данных

Основные сущности:

- `HelpRequest` — заявка на помощь;
- `Volunteer` — волонтер;
- `WeatherWarning` — погодное предупреждение;
- `NotificationMessage` — сообщение координатору;
- `RequestStatus`, `HelpType`, `LocationType` — справочные перечисления.

Модель данных согласована с CRUD-анализом в `docs/task2_1_dfd_crud.md`.

### 2.4 Моделирование работы приложения

Функциональное моделирование выполнено в виде DFD уровня 0, DFD уровня 1 и DFD уровня 2 для процессов назначения волонтера и проверки просрочек. Также подготовлены CRUD-матрица, спецификация функций и таблица согласования с информационной моделью.

### 3.2 Тестирование ПС

В проекте подготовлены JUnit-тесты:

- создание заявки со статусом «Новая»;
- назначение волонтера и переход в статус «В работе»;
- проверка просрочки и формирование уведомления;
- обновление счетчика назначений волонтера;
- проверка дат старше/младше 3 дней.

Команда запуска:

```bash
mvn test
```

Контрольный пример приемки:

| Шаг | Действие | Ожидаемый результат |
| --- | --- | --- |
| 1 | Создать заявку на уличную помощь | Заявка создана со статусом «Новая», при риске добавлено погодное предупреждение |
| 2 | Назначить доступного волонтера | Статус изменен на «В работе» |
| 3 | Нажать «Выполнить» | Статус изменен на «Выполнена», записана дата завершения |
| 4 | Создать заявку с датой создания старше 3 дней без волонтера и запустить проверку | Статус изменен на «Просрочена», создано уведомление |
