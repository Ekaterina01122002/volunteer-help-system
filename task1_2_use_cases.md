## 1. Роли и потребности

| Роль | Потребность | Основные действия |
| --- | --- | --- |
| Заявитель | Быстро оставить заявку и получить помощь | передать данные заявки оператору/координатору, уточнить статус вне интерфейса приложения |
| Волонтер | Видеть назначенную работу и завершать ее | получить назначение, выполнить заявку, сообщить о завершении |
| Координатор | Управлять потоком заявок | просматривать заявки, назначать волонтеров, проверять просрочки, получать уведомления |
| Администратор / оператор | Поддерживать корректность данных | вести справочник волонтеров, редактировать заявки, проверять журнал |

## 3. Сценарии использования

### UC-01. Создать заявку

**Актор:** оператор или координатор, действующий со слов заявителя.

**Предусловие:** приложение запущено, доступна вкладка «Заявки».

**Основной поток:**

1. Оператор или координатор получает данные от заявителя и нажимает «Добавить».
2. Система открывает форму заявки.
3. Пользователь вводит ФИО, телефон, тип помощи, тип места, адрес, дату, время и описание.
4. Система проверяет обязательные поля.
5. Система создает заявку со статусом «Новая».
6. Если место оказания помощи — улица, система проверяет погоду.
7. Система сохраняет заявку в JSON-хранилище.

**Альтернативные потоки:** если обязательные поля не заполнены, система не сохраняет заявку и сообщает об ошибке; если погодный API недоступен, используется учебная заглушка.

**Постусловие:** заявка отображается в таблице.

### UC-02. Назначить волонтера

**Актор:** координатор.

**Предусловие:** есть заявка в статусе «Новая» или «Просрочена», есть доступный волонтер.

**Основной поток:**

1. Координатор открывает вкладку «Назначение».
2. Система показывает заявки, доступные для назначения.
3. Координатор выбирает заявку и волонтера.
4. Координатор нажимает «Назначить волонтера».
5. Система записывает идентификатор волонтера в заявку.
6. Система переводит заявку в статус «В работе».
7. Система обновляет историю статусов и счетчик назначений волонтера.

**Альтернативные потоки:** если волонтер недоступен или заявка закрыта, система показывает ошибку.

**Постусловие:** заявка находится в работе.

### UC-03. Завершить заявку

**Актор:** волонтер или координатор.

**Предусловие:** заявка находится в статусе «В работе».

**Основной поток:**

1. Пользователь выбирает заявку в таблице.
2. Пользователь нажимает «Выполнить».
3. Система проверяет статус заявки.
4. Система переводит заявку в статус «Выполнена».
5. Система записывает дату завершения и событие в историю.

**Альтернативные потоки:** если заявка не находится в работе, система показывает ошибку.

**Постусловие:** заявка закрыта и недоступна для редактирования ключевых полей.

### UC-04. Проверить просроченные заявки

**Актор:** координатор.

**Предусловие:** в системе есть заявки.

**Основной поток:**

1. Координатор нажимает «Проверить просрочки».
2. Система выбирает заявки в статусе «Новая» без назначенного волонтера.
3. Система проверяет дату создания каждой заявки.
4. Если заявка старше 3 дней, система переводит ее в статус «Просрочена».
5. Система формирует сообщение координатору.
6. Система сохраняет изменения.

**Альтернативные потоки:** если просроченных заявок нет, система показывает количество `0`.

**Постусловие:** просроченные заявки отмечены, уведомления сформированы.

### UC-05. Управлять волонтерами

**Актор:** администратор / оператор.

**Предусловие:** приложение запущено, доступна вкладка «Волонтеры».

**Основной поток:**

1. Пользователь открывает вкладку «Волонтеры».
2. Пользователь добавляет, редактирует или удаляет запись волонтера.
3. Система проверяет ФИО и телефон.
4. Система сохраняет справочник.

**Альтернативные потоки:** если волонтер назначен на активную заявку, система запрещает удаление.

**Постусловие:** справочник волонтеров обновлен.
